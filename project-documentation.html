Project Overview

Secure File Vault is a full-stack web application that provides seven cryptographic and security-analysis tools in a single-page dashboard. It allows users to encrypt, decrypt, sign, hash, and inspect files entirely in their browser — with all heavy cryptographic work done server-side in Python.

🔒

File Encryption

Encrypt any file using AES-256-GCM with a password. Every encryption is unique thanks to a random salt and nonce.

🔓

File Decryption

Decrypt vault-encrypted files. Wrong passwords are clearly reported — no silent failures.

✍

Digital Signatures

Generate RSA-2048 key pairs, sign files, and verify whether a file has been tampered with.

⬡

Hash Generator

Compute SHA-256 and SHA-512 fingerprints of any file. Copy hashes to clipboard with one click.

🛡

Integrity Checker

Upload a file and its known-good hash. Confirms whether the file is original or has been modified.

📦

Compressed File Checker

Inspect ZIP and TAR archives without extracting them. Detects double extensions, nested archives, and path traversal attacks.

🕵

Obfuscated File Checker

Analyse filenames for obfuscation tricks (RTLO, double extensions, hidden characters). Returns a Low / Medium / High risk score.

Why these choices?

Flask was chosen because it is minimal, readable, and perfect for explaining every line of code in a presentation. There is no magic — each route is a plain Python function.

The cryptography library (by the Python Cryptographic Authority) is the gold-standard Python crypto library. It exposes low-level primitives correctly without hiding important details.

Vanilla JavaScript (no React, no jQuery) keeps the frontend simple enough to read in one sitting, while still providing a smooth user experience via the Fetch API.

Feature 1 — File Encryption

Encrypts any file using AES-256-GCM, the same algorithm used by TLS 1.3, WhatsApp, and Signal.

How It Works — Step by Step

1

Generate Random Salt & Nonce

A 16-byte salt and 12-byte nonce (number used once) are generated using the OS cryptographic random number generator. These are unique for every single encryption — even encrypting the same file with the same password produces a different output.

2

Derive the Encryption Key (PBKDF2-HMAC-SHA256)

The raw password is never used directly as a key. Instead, PBKDF2-HMAC-SHA256 runs 390,000 iterations to stretch the password into a 256-bit AES key. This makes brute-force attacks extremely slow.

3

Encrypt with AES-256-GCM

AES-256 is the cipher (block size 128-bit, key size 256-bit). GCM mode (Galois/Counter Mode) provides both confidentiality (nobody can read it) and authentication (any tampering is detected via the 16-byte tag appended to the ciphertext).

4

Save Output File

The output file layout is: [16-byte salt][12-byte nonce][ciphertext + 16-byte GCM tag]. Everything needed to decrypt is self-contained in the file — no separate metadata file is needed.

# modules/encryption.py (simplified)
def encrypt_file(input_path, output_path, password):
    salt  = os.urandom(16)   # random 16-byte salt
    nonce = os.urandom(12)   # random 12-byte nonce
    key   = derive_key(password, salt)  # PBKDF2 → 256-bit key

    aesgcm     = AESGCM(key)
    ciphertext = aesgcm.encrypt(nonce, plaintext, None)

    # Write: salt | nonce | ciphertext+tag
    output.write(salt + nonce + ciphertext)

🔐 Why AES-256-GCM?

GCM mode provides Authenticated Encryption with Associated Data (AEAD). If even one byte of the ciphertext is modified, the GCM tag verification fails and decryption is rejected. This prevents both eavesdropping and tampering.

Feature 2 — File Decryption

Decryption reverses the encryption process. The salt and nonce are read from the file header, the key is re-derived from the password, and AES-GCM decrypts and verifies the ciphertext.

⚠ Wrong Password Handling

If the password is wrong, AES-GCM's authentication tag will not match. The cryptography library raises an InvalidTag exception, which is caught and returned as a clear error message: "Incorrect password or the file has been tampered with." — no silent failures.

def decrypt_file(input_path, output_path, password):
    salt, nonce, ciphertext = parse_header(data)
    key = derive_key(password, salt)   # same PBKDF2 process

    try:
        plaintext = AESGCM(key).decrypt(nonce, ciphertext, None)
    except InvalidTag:
        raise ValueError("Incorrect password or tampered file.")

Feature 3 — Digital Signatures

Uses RSA-2048 asymmetric cryptography to prove a file is authentic and unmodified.

Three-Step Workflow

1

Generate Key Pair

An RSA-2048 key pair is generated. The private key is kept secret and used to sign. The public key can be shared freely and is used to verify. Keys are saved as PEM files in the keys/ folder.

2

Sign a File

The private key computes the SHA-256 digest of the file and encrypts it using PKCS#1v15 padding. The result is a signature — a blob of bytes that is mathematically tied to both the file content and the private key.

3

Verify Signature

Anyone with the public key can verify: the public key decrypts the signature and compares the result to the file's current SHA-256 digest. If they match → file is original. If not → file has been modified.

🔑 Asymmetric vs. Symmetric Cryptography

Encryption uses a symmetric key (same key encrypts and decrypts). Digital signatures use asymmetric keys: the private key signs (only the owner can do this), and the public key verifies (anyone can do this). This is the basis of TLS certificates, code signing, and email authentication.

Feature 4 — Hash Generator

Computes the SHA-256 and SHA-512 cryptographic hash of any file.

What is a Cryptographic Hash?

A hash function takes any input and produces a fixed-size output (the "digest") with three critical properties:

  • Deterministic — the same file always produces the same hash.
  • Avalanche effect — changing one bit of the file changes ~50% of the hash bits.
  • One-way — you cannot reverse a hash to recover the original file.
  • Collision-resistant — it is computationally infeasible to find two different files with the same hash.
SHA-256: 64 hex characters = 256 bits SHA-512: 128 hex characters = 512 bits

Files are read in 64 KB chunks so even multi-gigabyte files can be hashed without loading them entirely into memory.

Feature 5 — File Integrity Checker

The user provides a file and its original hash (obtained earlier or from a trusted source). The server recomputes the file's hash and compares it.

The algorithm (SHA-256 or SHA-512) is auto-detected from the hash length — 64 characters = SHA-256, 128 characters = SHA-512.

📌 Real-World Use Case

Software publishers (e.g., Linux distributions, Python.org) publish SHA-256 hashes of their downloads alongside the files. Users can verify the downloaded file hasn't been tampered with by comparing hashes — this is exactly what this tool does.

Feature 6 — Compressed File Checker

Inspects ZIP and TAR archives without extracting them, detecting threats that could harm the system.

Threats Detected

ThreatExampleWhy Dangerous
Double extension invoice.pdf.exe Windows shows "invoice.pdf" but runs the .exe
Executable extension setup.bat, run.vbs These files can execute system commands when run
Nested archive archive.zip inside archive.zip Can hide malicious payloads from one-level scanners
Path traversal ../../etc/passwd Extracting could overwrite files outside the target folder

Python's built-in zipfile and tarfile modules let us read the archive's table of contents (list of filenames + sizes) without decompressing a single byte of content.

Security Design Decisions

What the vault protects against

Password never stored or transmitted in plaintext

The user's password goes through PBKDF2-HMAC-SHA256 with 390,000 iterations. Only the derived 256-bit key is ever used. The password itself is discarded immediately after key derivation. The salt needed to re-derive the key is stored inside the encrypted file itself.

Every encryption is unique — no IV reuse

Both the salt and the nonce are generated from the OS random number generator (os.urandom) for every single encryption. This prevents nonce reuse attacks, which can catastrophically break AES-GCM if the same nonce is used twice with the same key.

Authenticated Encryption prevents tampering

AES-GCM appends a 16-byte authentication tag. Any modification to the ciphertext — even a single bit flip — causes decryption to fail with a cryptographic error. You cannot modify an encrypted file and have it decrypt successfully.

Secure filename handling

All uploaded filenames are sanitised with werkzeug.utils.secure_filename before being saved to disk. This prevents path traversal attacks where an attacker might upload a file named ../../etc/passwd.

⚠ Production considerations (for the semester presentation)

In a real production system, you would also add: rate limiting (to prevent abuse), automatic cleanup of the uploads/ and outputs/ folders, HTTPS-only serving, CSRF protection on forms, and private key storage in a Hardware Security Module (HSM) rather than a plain file.

How to Run the Project

Prerequisites

Python 3.9 or higher. No other system dependencies required.

Installation

# 1. Clone or download the project
cd secure-file-vault

# 2. Install Python dependencies
pip install -r requirements.txt

# 3. Start the Flask server
python main.py

The server starts on http://localhost:24331 (or whatever the PORT environment variable is set to). Open that address in your browser to see the dashboard.

requirements.txt

flask==3.0.3
cryptography==42.0.8
werkzeug==3.0.3

First-time usage tips

1

Generate RSA Keys first

Go to Digital Signature → Step 1 → Generate Keys before attempting to sign or verify any file.

2

Encrypt → Decrypt round-trip

Encrypt any file with a password, then upload the .enc file to the Decrypt panel with the same password to confirm it works.

3

Test the integrity checker

Use the Hash Generator to get a file's SHA-256. Copy it. Then use the Integrity Checker on the same file — it should say "unmodified". Modify a byte in the file and try again — it will say "MODIFIED".
